    monte carlo tree search

    markov decision processes

    Let's set up a simple model for the world and investigate how one might make optimal decisions. Our model will be discrete. Let's represent the set of all possible states in our model as \(\mathcal{S}\). For some state \(s \in \mathcal{S}\), let \(\mathcal{A}(s)\) be the set of all available actions in that state. Taking an action from one state leads us to a new state. For \(s, s' \in \mathcal{S}\) and \(a \in \mathcal{A}(s)\), Let \(\mathcal{T}(s, a, s')\) be the transition probability of reaching state \(s'\) after taking action \(a\) in state \(s\). Let \(\mathcal{R}(s, a, s')\) be the reward gained after such a transition. For simplicity, let's assume that every sequence of actions eventually reaches a terminal state (a state \(s\) such that \(\mathcal{A}(s)\) is the empty set) and hence there is no way to accumulate infinite reward. We call the tuple \((\mathcal{S}, \mathcal{A}, \mathcal{T}, \mathcal{R})\) a Markov Decision Process (MDP).

    An optimal decision rule is a mapping from every state \(s \in \mathcal{S}\) to an action in \(\mathcal{A}(s)\), such that follwing the optimal decision rule for every state you visit in a traversal maximizes the expected cumulative reward of the traversal. For an MDP like ours, this is simply a lookup table and we'll denote it as \(\pi^*(s)\). To us explain our optimal decision rule, we'll define associated functions called values and action values. The value of a state \(s\) is the expected cumulative reward from following our optimal decision rule starting from state \(s\). It is denoted by \(V^*(s)\). The action value of an action \(a\) in a state \(s\) is expected cumulative reward we get from following our optimal decision rule starting from state \(s\), provided we take action \(a\) first. It is denoted by \(Q^*(s, a)\). It should be relatively clear that

    \[ \pi^*(s) = \arg\max_{a \in \mathcal{A}(s)} Q(s, a) \]
    \[ V^*(s) = \max_{a \in \mathcal{A}(s)} Q(s, a) \]
    \[ Q^*(s, a) = \sum_{s' \in \mathcal{S}}\mathcal{T}(s, a, s')(\mathcal{R}(s, a, s') + V^*(s')) \]

    For your curiosity, this is the Bellman equation for MDPs.

    If we can compute \(Q^*(s, a)\) for every possible pair \((s, a)\) of state and action in our MDP, then we know the optimal decision rule. However, even from a glance, it should be clear that computing these action values is very hard. To know \(Q^*\), we must find \(V^*\). But to find \(V^*\), we must know \(Q^*\)! If your interested in one way to solve this problem, I encourage you to look at the Value Iteration Algorithm. However, value iteration requires you to loop over every single tuple \((s, a, s')\) of state, action, and resulting state in the entire MDP many times. For larger MDPs, the computation time is massive. Furthermore, you must wait for the algorithm to finish entirely before reasonably concluding even a single optimal decicion rule. This is because value iteration solves the MDP entirely, finding every optimal decision rule simultaneously.

    what is monte carlo tree search?

    Monte Carlo tree search is an algorithm for finding the optimal decision rule for a given starting state \(s\). To make the computation tractable, the algorithm takes two major shortcuts:

    1. The algorithm only computes the action values for states and actions which you can actually reach from the starting state \(s\), hence why the algorithm is formally known as a tree search. For many MDPs, this cuts down the number of values we need to compute immensely.
    2. The algorithm uses random sampling to estimate exptectations of cumulative reward, hence why this algorithm is formally knows as a Monte Carlo tree search. In general, Monte-Carlo methods use random sampling to estimate complex expectations which are intractable to solve exactly.

    To explain, let's suppose that we know the optimal decision rule for our MDP. Then the Monte Carlo way to compute \(Q^*(s, a)\) would be to start at state \(s\) and take action \(a\). Then, keep taking actions accoring to the optimal decision rule until termination, keeping track of the total accumulated reward. Repeat this process of running a full simulation hundreds, or thousands, or millions of times. The average total accumulated reward over all simulations converges to the true value of \(Q^*(s, a)\).

    But we don't know the optimal decision rule! How can we randomly sample from a distribution that we don't actually know yet? We need to devise a new sampling strategy which may start off wrong, but eventually corrects it's course and converges to the correct one. The correct way to sample is: "at every step choose the action which grants us the highest total expected cumulative reward". The sampling method that we will construct will be, informally, "choose the action based on a strategy which gets us as close to the highest total expected cumulative reward without knowing which action grants us this upfront". As the number of trials increase, this strategy will eventually always pick the optmial action, and hence we still converge to the correct value. Most importanly, this strategy minimizes the error on its way to convergence. We borrow this strategy from a famous problem in reinforcement learning.

    The multi-armed bandit problem

    Suppose that you have a large number of coins and you enter a room with multiple slot machines. You know nothing about the expected ayout of the slot machines. Some slot machines could have higher expected payouts than other slot machines. How do you maximize your expected total payout? Well, we wan't to play the slot machine with the highest expected payout as mant times as possible!

    At a given step, we can compute the empirical average of the payout each slot machine observed so far. A reasonable strategy would be to choose the slot machine which the highest average payout, right? But if we have only played a certain slot machine a small number of times, then the average payout might be far from the expected payout. Lets devise a better metric than the average. At a given step, set some probability threshold \(p\), which should be small. Then, for each slot machine, compute an estimate for the expected payout of said slot machine such that the probability of the true expected payout being higher than the estimate is at most \(p\). Therefore if we reject a slot machine based on this estimate, the probability of falsely rejecting that slot machine is at most \(p\).

    We can construct such an estimate using Hoeffding's Inequality. Let \(X_1, X_2, \ldots, X_n\) be a sample of \(n\) payouts from a given slot machine. Let \(\overline{X}\) be thier average and \(\mathbb{E}[X]\) be the true expected payout. Finally, assume that each payout is bounded in the range \([a, b]\). Then

    \[ \mathbb{P}(\mathbb{E}[X] > \overline{X} + t) \leq \exp\left(-\frac{2nt^2}{(b - a)^2}\right) \]

    If we set the right hand side equal to \(p\) and solve for \(t\) we get

    \[ t = (b - a)\sqrt{\frac{-\ln(p)}{2n}} \]

    and now we have the estimate we are looking for.

    Finally, we need a way to set the value of \(p\). Recall that \(p\) is the bound on the probabilty of falsly rejecting a slot machine. This should relate to the number of tokens we have spent. If we have only spent a few tokens, then the total uncertainty is high and it is okay if we falsely reject a slot machine to make room for one which has proved more lucrative so far. However, if we have spent a large number of tokens, we want to be absolutely sure that we have found the true optimal slot machine. A paper called Finite-time Analysis of the Multiarmed Bandit Problem proved that setting \(p = N^{-4}\), where \(N\) is the number of tokens spend so far, is best, and that the resuting estimate

    \[ UCB_i = \overline{X}_i + (b - a)\sqrt{\frac{2\ln(N)}{n_i}} \]

    called the Upper Confidence Bound (UCB), yields the optimal algorithm for maximizing profit.

    the monte carlo tree search algorithm

    To find the optimal action from a starting state \(s\), Monte Carlo Tree Search treats every state in the MDP as its own independent multi-armed bandit problem. The algorithm iteratively loops through two main steps to build a localized decision tree:

    1. Sampling: Starting from \(s\), we traverse a continuous path to a terminal state by always choosing the action that maximizes the UCB estimate. Crucially, if an action has never been tried (\(n_i=0\)), its UCB estimate is mathematically infinite. Therefore, we will always pick an unexplored action over an explored one. Once we select an unexplored action, we have no empirical data to compute UCB estimates for the subsequent states, so we simply complete the rest of the trajectory to the terminal state by picking actions entirely at random.
    2. Updating: Once the terminal state is reached and the total accumulated reward is calculated, we walk back up the exact path we just sampled. For every state-action pair we visited, we increment its visit count \(n_i\) and update its empirical average \(\overline{X}_i\) using the new reward.

    We repeat this loop until our computational budget is exhausted. Finally, we look at the actions available from our root state \(s\) and select the one with the highest empirical average (or the highest visit count) as our optimal move.

    things about rust

    Known in the community as "The Book", I am reading the The Rust Programming Language, the official introductory textbook for Rust, partly because I wanted to code my chess engine in the language, and as silly as it sounds, I wanted to get back into reading books in general (yes, I purchased the paper copy on Amazon).

    I don't want to become a Rust cultist who's life mission is to tell everybody that Rust is the only programming language you should use for anything more than "Hello, World!". I'll be frank and say that for a student/hobbiest, the insane amount of rules imposed by the Rust compiler is a little bit annoying. I want you to let me write slightly unsafe code when I want to. But hey, that's what garbage collected languages are for. I'm writing this from the perspective of somebody who doesn't write low-level code very often. For systems programming, I do believe that this is the future.

    Either way, I want a taste of what Rust is like. In this post I want to list some of the things about Rust that I do really like, or at least find interesting. This post is not about the things I don't like. Maybe I'll add that later on. Finally, I won't include too much about the borrow checker, or else the list would be too long. I highly recommend you learn about it yourself. Finally, this list is in progress. I will add to it as I continue through the book and onwards.

    Things I like (or find interesting) about the Rust programming language:

    • Immutability as a default! My code tells the reader exactly when a variable is expected to change, and when not!
    • Type inferencing: the way the compiler looks all over your code and decides what should be the type in every situation. Take a look at three ways to parse a string into a byte. In Rust, you use the .parse() method on a string to convert it into another type. It's a generic method, so it can output any type, so long as it has the FromStr trait. A byte, or u8 has this trait, and we can convert a string to a byte with:
    let parsed = "42".parse::<u8>().unwrap();

    We pass the type u8 directly into the function to tell it what to use. To clean up that syntax, the compiler can look backwards in the code to infer what type it should use automatically. For example:

    let parsed: u8 = "42".parse().unwrap();

    This code works exactly the same. But what's really cool, the compiler can look forwards in your code too. Here's another snippet that works just the same:

    fn main() {
    let parsed = "42".parse().unwrap();
    print_byte(parsed);
}

fn print_btye(byte: u8) {
    println!("My byte is {}!", byte);
}

    The compiler looked ahead and saw that the value is used in a function that takes in a byte, and inferred that a byte is what we wanted to convert the string to!

    • Labeling loops, and being able to break out of or continue outer loops in a nested loop from the inner one. 
      'outer_loop loop {
    ... // some code
    loop {
        ... // some code
        if condition {
            break 'outer_loop;
        }
    }
}
    • Disallowing the existence of ANY other references to a value as long as a mutable reference already exists. This just makes sense!
    • The way that references to values have dynamic scopes. The scope of a reference starts when it is defined, like any variable. But thier scope ends exactly at the last line they are used! This is different from regular variables, which have scope which last until the end of the curly braces.

    chess, again

    If you've known about my interests in programming, you've known I've always been obsessed with programming a chess engine from scratch. In 2023, I leared C++ just so I could give it a shot, and I loved every second of optimizing every line of code to run as fast as I could. Of course I went for the old fasioned approach of "crunch as many positions as you can" for finding the best move.

    Whether I knew it or not at the time, I am aware that for an engine stategy like that, speed is must less important than smart things such as better heuristics for pruning more branches of the search tree. I was more interested in making things fast over making them smart.

    I often fall into the pitfall of premature optimization, and while it's fun, I know now how damaging it can really be. I think now my focus has shifted more towards writing clean code over hyper optimized code. I spend too much time thinking about how my code looks, or how "professional" it is. It's a major time waster but hey, I program for fun, and the end result is always worth it to me.

    I think at about the time my paradigm shifted I decided to completely rewrite my original chess engine. I changed the core design principals. My original attempt was completely object oriented and full of encapsulation. I had just taken the class "Object Oriented Programming" in college. It took me about a year to realize that I despised it. I has also started to get really into legacy C code. I adopted procedural programming with structs and I found it greatly satisfying. I did feel like a professional and I felt in control. With this new control, along with some classy improvements like bitboards, I found some serious improvements. However, I got busy with school and never really finished the job. Although I still really love that code I wrote.

    As I stepped back into the modern world, I started to get really into fancy tooling. Programming in Python forced me to adopt modern workflows, and I got realy addicted. Also, as much as I loved my second iteration, I used CMake and, god, I HATED IT. It all felt ancient. And of course, my main gig is Python and so I tried to incorporate it into my chess project: building and training neural networks that I could implement into my engine. But wrangling with ctypes also absolutely sucked. So now that I have time to get back into this project, as much as I love the code that I wrote the second time around, I just can't see myself getting back into it.

    This brings me to today. As crazy as it sounds, I'm gonna rewite my engine from scratch for the SECOND TIME. But there are several things that I am going to do different. First, I'm gonna stick to what I am really interested in, machine learning. This'll heavily rely on deep neural networks to do the heavy lifting. So although I do want a fast lower layer, I am not going to kill myself with optimizations. Second, to satisfy my modern tastes, I am going to try out a fancy new modern language with lots of buzz, Rust. It looks realy attractive to me. First of all, cargo looks fantastic. And with things like PyO3 and maturin, I am GIDDY with the anticipation of baking this directly into my favorite langauage, Python. Second of all, the language's strict rules really tickle my obsession with writing clean, professional code. And its speed with zero-cost abstractions makes my incessecant premature optimization devil happy.

    I've been obsessed with this AlphaZero paper by Google, and so my goal is to recreate it myself. I just got a fancy new M5 macbook pro, and so I really want to see how far I can take it by sticking to my hardware alone. I won't talk too much about it today, since I want to document the project on this blog. I just wanted to describe my journey up to this point.
